# jq দিয়ে Hayabusa-এর ফলাফল বিশ্লেষণ

# লেখক

Zach Mathis ([@yamatosecurity](https://twitter.com/yamatosecurity)) - 2023/03/22

# সম্পর্কে

লগের গুরুত্বপূর্ণ ফিল্ডগুলো শনাক্ত করা, বের করে আনা এবং সেগুলোর উপর ভিত্তি করে মেট্রিক তৈরি করা DFIR এবং থ্রেট হান্টিং বিশ্লেষকদের জন্য একটি অপরিহার্য দক্ষতা।
টাইমলাইন বিশ্লেষণের জন্য Excel বা Timeline Explorer-এর মতো প্রোগ্রামে আমদানি করার উদ্দেশ্যে Hayabusa-এর ফলাফল সাধারণত `.csv` ফাইলে সংরক্ষণ করা হয়।
তবে, যখন শত শত বা তারও বেশি একই ধরনের ইভেন্ট থাকে, তখন সেগুলো ম্যানুয়ালি যাচাই করা অবাস্তব বা অসম্ভব হয়ে পড়ে।
এই পরিস্থিতিতে, বিশ্লেষকরা সাধারণত একই ধরনের ডেটা সাজিয়ে ও গণনা করে অস্বাভাবিকতা (outlier) খোঁজেন।
এটি long tail analysis, stack ranking, frequency analysis ইত্যাদি নামেও পরিচিত।
Hayabusa দিয়ে ফলাফল `.json` বা `.jsonl` ফাইলে আউটপুট করে এবং তারপর `jq` দিয়ে বিশ্লেষণ করে এটি করা যায়।

উদাহরণস্বরূপ, একজন বিশ্লেষক একটি প্রতিষ্ঠানের সব ওয়ার্কস্টেশনে ইনস্টল করা সার্ভিসগুলো তুলনা করতে পারেন।
যদিও কোনো নির্দিষ্ট ম্যালওয়্যার প্রতিটি ওয়ার্কস্টেশনে ইনস্টল হওয়া সম্ভব, তবে সম্ভাবনা বেশি যে এটি কেবল কয়েকটি সিস্টেমেই থাকবে।
এই ক্ষেত্রে, সব সিস্টেমে ইনস্টল করা সার্ভিসগুলো নিরীহ হওয়ার সম্ভাবনা বেশি, অন্যদিকে বিরল সার্ভিসগুলো বেশি সন্দেহজনক হতে থাকে এবং সেগুলো পর্যায়ক্রমে যাচাই করা উচিত।

আরেকটি ব্যবহারের ক্ষেত্র হলো কোনো কিছু কতটা সন্দেহজনক তা নির্ধারণে সাহায্য করা।
উদাহরণস্বরূপ, একজন বিশ্লেষক `4625` ব্যর্থ লগন লগ বিশ্লেষণ করে নির্ধারণ করতে পারেন যে একটি নির্দিষ্ট IP ঠিকানা কতবার লগন করতে ব্যর্থ হয়েছে।
যদি কেবল কয়েকটি ব্যর্থ লগন থাকে, তবে সম্ভবত একজন প্রশাসক কেবল তার পাসওয়ার্ড ভুল টাইপ করেছিলেন।
তবে, যদি একটি নির্দিষ্ট IP ঠিকানা থেকে অল্প সময়ের মধ্যে শত শত বা তারও বেশি ব্যর্থ লগন হয়, তবে সম্ভবত সেই IP ঠিকানাটি ক্ষতিকর।

`jq` কীভাবে ব্যবহার করতে হয় তা শেখা আপনাকে কেবল Windows ইভেন্ট লগ নয়, বরং সব JSON ফরম্যাটেড লগ বিশ্লেষণে দক্ষ হতে সাহায্য করবে।
এখন যেহেতু JSON একটি অত্যন্ত জনপ্রিয় লগ ফরম্যাটে পরিণত হয়েছে এবং বেশিরভাগ ক্লাউড প্রোভাইডার তাদের লগের জন্য এটি ব্যবহার করে, `jq` দিয়ে সেগুলো পার্স করতে পারা আধুনিক নিরাপত্তা বিশ্লেষকের জন্য একটি অপরিহার্য দক্ষতা হয়ে উঠেছে।

এই গাইডে, যারা আগে কখনো `jq` ব্যবহার করেননি তাদের জন্য আমি প্রথমে কীভাবে এটি ব্যবহার করতে হয় তা ব্যাখ্যা করব এবং তারপর বাস্তব জগতের উদাহরণসহ আরও জটিল ব্যবহার ব্যাখ্যা করব।
`jq`-কে `sort`, `uniq`, `grep`, `sed` ইত্যাদির মতো অন্যান্য দরকারি কমান্ডের সাথে সংযুক্ত করতে পারার জন্য আমি linux, macOS বা Windows-এ linux ব্যবহারের পরামর্শ দিই।

# jq ইনস্টল করা

দয়া করে [https://stedolan.github.io/jq/](https://stedolan.github.io/jq/) দেখুন এবং `jq` কমান্ডটি ইনস্টল করুন।

# JSON ফরম্যাট সম্পর্কে

JSON লগ হলো কোঁকড়ানো বন্ধনী `{` `}`-এর মধ্যে থাকা অবজেক্টের একটি তালিকা।
এই অবজেক্টগুলোর ভিতরে কোলন দিয়ে পৃথক করা key-value জোড়া থাকে।
key অবশ্যই স্ট্রিং হতে হবে, তবে value নিচের যেকোনো একটি হতে পারে:
  * স্ট্রিং (উদাহরণ: `"string"`)
  * সংখ্যা (উদাহরণ: `10`)
  * আরেকটি অবজেক্ট (উদাহরণ: `{ xxxx }`)
  * অ্যারে (উদাহরণ: `["string", 10]`)
  * বুলিয়ান (উদাহরণ: `true`, `false`)
  * `null`

আপনি অবজেক্টের ভিতরে যত খুশি অবজেক্ট নেস্ট করতে পারেন।

এই উদাহরণে, `Details` হলো একটি রুট অবজেক্টের ভিতরে একটি নেস্টেড অবজেক্ট:
```
{
    "Timestamp": "2016-08-19 08:06:57.658 +09:00",
    "Computer": "IE10Win7",
    "Channel": "Sec",
    "EventID": 4688,
    "Level": "info",
    "RecordID": 6845,
    "RuleTitle": "Proc Exec",
    "Details": {
        "CmdLine": "C:\\Windows\\system32\\ipconfig /release",
        "Path": "C:\\Windows\\System32\\ipconfig.exe",
        "PID": "0xcf4",
        "User": "IE10WIN7$",
        "LID": "0x3e7"
    }
}
```

# Hayabusa-এর সাথে JSON এবং JSONL ফরম্যাট সম্পর্কে

পূর্ববর্তী সংস্করণগুলোতে, Hayabusa সব `{ xxx }` লগ অবজেক্টকে একটি বিশাল অ্যারের মধ্যে রাখার ঐতিহ্যবাহী JSON ফরম্যাট ব্যবহার করত।

উদাহরণ:
```
[
    {
        "Timestamp": "2016-08-19 08:06:57.658 +09:00",
        "Computer": "IE10Win7",
        "Channel": "Sec",
        "EventID": 4688,
        "Level": "info",
        "RecordID": 6845,
        "RuleTitle": "Proc Exec",
        "Details": {
            "CmdLine": "C:\\Windows\\system32\\ipconfig /release",
            "Path": "C:\\Windows\\System32\\ipconfig.exe",
            "PID": "0xcf4",
            "User": "IE10WIN7$",
            "LID": "0x3e7"
        }
    },
    {
        "Timestamp": "2016-08-19 11:07:47.489 +09:00",
        "Computer": "IE10Win7",
        "Channel": "Sec",
        "EventID": 4688,
        "Level": "info",
        "RecordID": 6847,
        "RuleTitle": "Proc Exec",
        "Details": {
            "CmdLine": "taskhost.exe $(Arg0)",
            "Path": "C:\\Windows\\System32\\taskhost.exe",
            "PID": "0x228",
            "User": "IE10WIN7$",
            "LID": "0x3e7"
        }
    }
]
```

এর দুটি সমস্যা আছে।
প্রথম সমস্যাটি হলো `jq` কোয়েরিগুলো আরও জটিল হয়ে যায় কারণ সবকিছু একটি অতিরিক্ত `.[]` দিয়ে শুরু করতে হয় যাতে এটিকে সেই অ্যারের ভিতরে দেখতে বলা যায়।
আরও বড় সমস্যাটি হলো এমন লগ পার্স করার জন্য কোনো কিছুর প্রয়োজন হলে প্রথমে অ্যারের সব ডেটা লোড করা আবশ্যক।
আপনার যদি খুব বড় JSON ফাইল থাকে এবং পর্যাপ্ত মেমরি না থাকে তবে এটি একটি সমস্যা হয়ে দাঁড়ায়।
প্রয়োজনীয় CPU এবং মেমরি ব্যবহার কমানোর জন্য, JSONL (JSON Lines) ফরম্যাট, যা সবকিছুকে একটি বিশাল অ্যারের মধ্যে রাখে না, আরও জনপ্রিয় হয়ে উঠেছে।
Hayabusa JSON এবং JSONL ফরম্যাটে আউটপুট দেয়, তবে JSON ফরম্যাট আর কোনো অ্যারের ভিতরে সংরক্ষণ করা হয় না।
পার্থক্য কেবল এটুকু যে JSON ফরম্যাট একটি টেক্সট এডিটরে বা কনসোলে পড়া সহজ, অন্যদিকে JSONL ফরম্যাট প্রতিটি JSON অবজেক্ট একটি একক লাইনে সংরক্ষণ করে।
JSONL ফরম্যাট কিছুটা দ্রুততর এবং আকারে ছোট হবে তাই এটি আদর্শ যদি আপনি লগগুলো কেবল একটি SIEM ইত্যাদিতে আমদানি করতে যাচ্ছেন কিন্তু সেগুলো দেখতে যাচ্ছেন না।
JSON ফরম্যাট আদর্শ যদি আপনি কিছু ম্যানুয়াল যাচাইও করতে যাচ্ছেন।

# JSON ফলাফল ফাইল তৈরি করা

Hayabusa-এর বর্তমান 2.x সংস্করণে, আপনি ফলাফল `hayabusa json-timeline -d <directory> -o results.json` দিয়ে JSON-এ বা JSONL ফরম্যাটের জন্য `hayabusa json-timeline -d <directory> -J -o results.jsonl` দিয়ে সংরক্ষণ করতে পারেন।

Hayabusa ডিফল্ট `standard` প্রোফাইল ব্যবহার করবে এবং `Details` অবজেক্টে বিশ্লেষণের জন্য কেবল ন্যূনতম পরিমাণ ডেটা সংরক্ষণ করবে।
আপনি যদি .evtx লগের সব মূল ফিল্ড তথ্য সংরক্ষণ করতে চান, তবে আপনি `--profile all-field-info` অপশনসহ `all-field-info` প্রোফাইলটি ব্যবহার করতে পারেন।
এটি সব ফিল্ড তথ্য `AllFieldInfo` অবজেক্টে সংরক্ষণ করবে।
আপনি যদি নিরাপত্তার জন্য `Details` এবং `AllFieldInfo` উভয় অবজেক্টই সংরক্ষণ করতে চান, তবে আপনি `super-verbose` প্রোফাইলটি ব্যবহার করতে পারেন।

## AllFieldInfo-এর পরিবর্তে Details ব্যবহারের সুবিধা

`AllFieldInfo`-এর পরিবর্তে `Details` ব্যবহারের প্রথম সুবিধা হলো কেবল গুরুত্বপূর্ণ ফিল্ডগুলোই সংরক্ষণ করা হয়, এবং ফাইলের জায়গা বাঁচাতে ফিল্ডের নামগুলো সংক্ষিপ্ত করা হয়েছে।
এর অসুবিধা হলো এমন ডেটা মিস হওয়ার সম্ভাবনা থাকে যা আসলে আপনার কাছে গুরুত্বপূর্ণ ছিল কিন্তু বাদ পড়ে গিয়েছিল।
দ্বিতীয় সুবিধা হলো Hayabusa ফিল্ডের নাম স্বাভাবিক (normalize) করে আরও সমরূপভাবে ফিল্ডগুলো সংরক্ষণ করবে।
উদাহরণস্বরূপ, মূল Windows লগে, ব্যবহারকারীর নাম সাধারণত একটি `SubjectUserName` বা `TargetUserName` ফিল্ডে থাকে।
তবে, কখনো কখনো ব্যবহারকারীর নাম একটি `AccountName` ফিল্ডে থাকবে, কখনো টার্গেট ব্যবহারকারী আসলে `SubjectUserName` ফিল্ডে থাকবে, ইত্যাদি...
দুর্ভাগ্যবশত, Windows ইভেন্ট লগে অনেক অসামঞ্জস্যপূর্ণ ফিল্ডের নাম আছে।
Hayabusa এই ফিল্ডগুলো স্বাভাবিক করার চেষ্টা করে, যাতে একজন বিশ্লেষককে Windows-এ ইভেন্ট আইডিগুলোর মধ্যে অসীম পরিমাণ খুঁটিনাটি ও অসঙ্গতি বোঝার পরিবর্তে কেবল একটি সাধারণ নাম পার্স করতে হয়।

এখানে user ফিল্ডের একটি উদাহরণ দেওয়া হলো।
Hayabusa `SubjectUserName`, `TargetUserName`, `AccountName` ইত্যাদিকে নিম্নলিখিতভাবে স্বাভাবিক করবে:
  * `SrcUser` (Source User): যখন কোনো ব্যবহারকারীর **থেকে** একটি অ্যাকশন ঘটে। (সাধারণত একজন রিমোট ব্যবহারকারী।)
  * `TgtUser` (Target User): যখন কোনো ব্যবহারকারীর **প্রতি** একটি অ্যাকশন ঘটে। (উদাহরণস্বরূপ, একজন ব্যবহারকারীর **প্রতি** একটি লগন।)
  * `User`: যখন বর্তমানে লগ ইন করা একজন ব্যবহারকারী দ্বারা একটি অ্যাকশন ঘটে। (অ্যাকশনে কোনো নির্দিষ্ট দিক নেই।)

আরেকটি উদাহরণ হলো প্রসেস।
মূল Windows ইভেন্ট লগে, প্রসেস ফিল্ডকে একাধিক নামকরণ রীতিতে উল্লেখ করা হয়: `ProcessName`, `Image`, `processPath`, `Application`, `WindowsDefenderProcessName` ইত্যাদি...
ফিল্ড নরমালাইজেশন ছাড়া, একজন বিশ্লেষককে প্রথমে সব ভিন্ন ভিন্ন ফিল্ডের নাম সম্পর্কে জ্ঞাত হতে হবে, তারপর এই ফিল্ডের নামসহ সব লগ বের করতে হবে, তারপর সেগুলো একসাথে সংযুক্ত করতে হবে।

একজন বিশ্লেষক Hayabusa-এর `Details` অবজেক্টে প্রদান করা স্বাভাবিক করা একক `Proc` ফিল্ডটি ব্যবহার করে অনেক সময় ও ঝামেলা বাঁচাতে পারেন।

# jq পাঠ/রেসিপি

আমি এখন বেশ কয়েকটি ব্যবহারিক উদাহরণের পাঠ/রেসিপি তালিকাভুক্ত করব যা আপনার কাজে সাহায্য করতে পারে।

## 1. jq এবং Less দিয়ে রঙে ম্যানুয়াল যাচাই

লগে কোন কোন ফিল্ড আছে তা বুঝতে এটি প্রথম কাজগুলোর একটি।
আপনি সহজভাবে একটি `less results.json` করতে পারেন কিন্তু একটি ভালো উপায় হলো নিম্নলিখিতটি:
`cat results.json | jq -C | less -R`

`jq`-তে পাস করার মাধ্যমে, এটি সব ফিল্ড পরিপাটিভাবে ফরম্যাট করবে যদি সেগুলো শুরুতে পরিপাটিভাবে ফরম্যাট করা না থাকে।
`jq`-এর সাথে `-C` (color) অপশন এবং `less`-এর সাথে `-R` (raw output) অপশন ব্যবহার করে, আপনি রঙে উপরে ও নিচে স্ক্রল করতে পারেন।

## 2. মেট্রিক

Hayabusa-এর ইতিমধ্যেই ইভেন্ট আইডির উপর ভিত্তি করে ইভেন্টের সংখ্যা ও শতকরা হার মুদ্রণ করার কার্যকারিতা আছে, তবে এটি `jq` দিয়ে কীভাবে করতে হয় তা জানাও ভালো।
এটি আপনাকে যে ডেটার জন্য মেট্রিক তৈরি করতে চান তা কাস্টমাইজ করতে দেবে।

আসুন প্রথমে নিম্নলিখিত কমান্ড দিয়ে Event ID-এর একটি তালিকা বের করি:

`cat results.json | jq '.EventID'`

এটি প্রতিটি লগ থেকে কেবল Event ID সংখ্যাটি বের করবে।
`jq`-এর পরে, একক উদ্ধৃতিতে, কেবল একটি `.` এবং আপনি যে ফিল্ডটি বের করতে চান তার নাম টাইপ করুন।
আপনি এইরকম একটি দীর্ঘ তালিকা দেখতে পাবেন:

`cat results.json | jq '.EventID' | sort | uniq -c`

`uniq`-এর জন্য `-c` অপশনটি একটি অনন্য ইভেন্ট আইডি কতবার ঘটেছে তা গণনা করবে।

 আপনি দেখতে পাচ্ছেন যে এটি সাজানো নয়, তাই কোন ইভেন্ট আইডিগুলো সবচেয়ে বেশি ঘটেছে তা বলা কঠিন।

 আপনি এটি ঠিক করতে শেষে একটি `sort -n` যোগ করতে পারেন:

`cat results.json | jq '.EventID' | sort | uniq -c | sort -n`

`-n` অপশনটি `sort`-কে সংখ্যা অনুসারে সাজাতে বলে।

দ্বিতীয় সর্বাধিক রেকর্ড করা ইভেন্ট ছিল `4625` (Failed Logon)।

আপনি যদি সবচেয়ে বেশি রেকর্ড করা ইভেন্টগুলো উপরে মুদ্রণ করতে চান, তবে আপনি `sort -n -r` বা `sort -nr` দিয়ে সাজানো উল্টে দিতে পারেন।
আপনি ফলাফল `head -n 10`-এ পাইপ করে কেবল শীর্ষ ১০টি সর্বাধিক রেকর্ড করা ইভেন্টও মুদ্রণ করতে পারেন।

## 3. নির্দিষ্ট ডেটার উপর ফিল্টার করা

অনেক সময় আপনি নির্দিষ্ট Event ID, ব্যবহারকারী, প্রসেস, LID (Logon ID) ইত্যাদির উপর ফিল্টার করতে চাইবেন...
আপনি `jq` কোয়েরির ভিতরে `select` দিয়ে তা করতে পারেন।

উদাহরণস্বরূপ, আসুন সব `4624` সফল লগন ইভেন্ট বের করি:

`cat results.json | jq 'select ( .EventID == 4624 ) '`

তবে একটি সমস্যা আছে।
আপনি `jq: error (at <stdin>:10636): Cannot index string with string "Type"` বলে ত্রুটি লক্ষ্য করতে পারেন।
যেকোনো সময় আপনি `Cannot index string with string` ত্রুটি দেখলে, এর অর্থ হলো আপনি `jq`-কে এমন একটি ফিল্ড আউটপুট করতে বলছেন যা বিদ্যমান নেই বা ভুল টাইপের।
আপনি ফিল্ডের শেষে একটি `?` যোগ করে এই ত্রুটিগুলো থেকে মুক্তি পেতে পারেন।
এটি `jq`-কে ত্রুটিগুলো উপেক্ষা করতে বলে।

উদাহরণ: `cat results.json | jq 'select ( ( .EventID == 4624 ) and ( .Details.Type? == 3 ) ) '`

এখন, নির্দিষ্ট মানদণ্ডের উপর ফিল্টার করার পরে, আমরা এখন আগ্রহের নির্দিষ্ট ফিল্ড নির্বাচন করতে `jq` কোয়েরির ভিতরে একটি `|` ব্যবহার করতে পারি।

উদাহরণস্বরূপ, আসুন টার্গেট ব্যবহারকারীর নাম `TgtUser` এবং উৎস IP ঠিকানা `SrcIP` বের করি:

`cat results.json | jq -j 'select ( ( .EventID == 4624 ) and ( .Details.Type? == 3 ) ) | .Details.TgtUser , " " , .Details.SrcIP , "\n" '`

আবার, আমরা একাধিক ফিল্ড আউটপুট করতে নির্বাচন করার জন্য `jq`-তে `-j` (join) অপশন যোগ করি।
তারপর আপনি পূর্ববর্তী উদাহরণগুলোর মতো `sort`, `uniq -c` ইত্যাদি চালাতে পারেন... একটি নির্দিষ্ট IP ঠিকানা একটি টাইপ 3 নেটওয়ার্ক লগনের মাধ্যমে কোনো ব্যবহারকারীতে কতবার লগ ইন করেছে তা জানতে।

## 4. CSV ফরম্যাটে আউটপুট সংরক্ষণ করা

দুর্ভাগ্যবশত, Windows ইভেন্ট লগের ফিল্ডগুলো ইভেন্টের ধরন অনুসারে সম্পূর্ণ ভিন্ন হবে, তাই শত শত কলাম ছাড়া ফিল্ড অনুযায়ী কমা-পৃথক করা টাইমলাইন তৈরি করা সহজভাবে সম্ভব নয়।
তবে, একক ধরনের ইভেন্টের জন্য ফিল্ড-পৃথক করা টাইমলাইন তৈরি করা সম্ভব।
দুটি সাধারণ উদাহরণ হলো lateral movement এবং password guessing/spraying যাচাই করার জন্য Security `4624` (সফল লগন) এবং `4625` (ব্যর্থ লগন)।

এই উদাহরণে, আমরা কেবল Security 4624 লগ বের করছি এবং timestamp, computer name এবং সব `Details` তথ্য আউটপুট করছি।
আমরা `| @csv` ব্যবহার করে এটি একটি CSV ফাইলে সংরক্ষণ করি, তবে, আমাদের ডেটা একটি অ্যারে হিসেবে পাস করতে হবে।
আমরা আগের মতো আউটপুট করতে চাওয়া ফিল্ডগুলো নির্বাচন করে এবং সেগুলোকে অ্যারেতে পরিণত করতে `[ ]` বর্গাকার বন্ধনী দিয়ে আবদ্ধ করে তা করতে পারি।

উদাহরণ: `cat results.json | jq 'select ( (.Channel == "Sec" ) and ( .EventID == 4624 ) ) | [ .Timestamp , .Computer , .Details[]? ] | @csv ' -r`

নোট:
  * `Details` অবজেক্টের সব ফিল্ড নির্বাচন করতে আমরা `[]` যোগ করি।
  * এমন কিছু ক্ষেত্র আছে যেখানে `Details` একটি স্ট্রিং, অ্যারে নয় এবং `Cannot iterate over string` ত্রুটি দেবে তাই আপনাকে একটি `?` যোগ করতে হবে।
  * ডাবল কোট ব্যাকস্ল্যাশ দিয়ে এস্কেপ না করার জন্য আমরা `jq`-তে `-r` (Raw output) অপশন যোগ করি।

আমরা যদি কেবল কারা সফল লগন করেছে তা যাচাই করছি, তবে আমাদের শেষ `LID` (Logon ID) ফিল্ডের প্রয়োজন নাও হতে পারে।
আপনি `del` ফাংশন দিয়ে যেকোনো অপ্রয়োজনীয় কলাম মুছে ফেলতে পারেন।

উদাহরণ: `cat results.json | jq 'select ( ( .Channel == "Sec" ) and ( .EventID == 4624 ) ) | [ .Timestamp , .Computer , .Details[]? ] | del( .[6] ) | @csv ' -r`

অ্যারে `0` থেকে গণনা করে তাই ৭ম ফিল্ডটি সরাতে, আমরা `6` ব্যবহার করি।

আপনি এখন `> 4624-logs.csv` যোগ করে CSV ফাইলটি সংরক্ষণ করতে পারেন এবং তারপর আরও বিশ্লেষণের জন্য এটি Excel বা Timeline Explorer-এ আমদানি করতে পারেন।

লক্ষ্য করুন যে ফিল্টারিং করতে আপনাকে একটি হেডার যোগ করতে হবে।
যদিও `jq` কোয়েরির ভিতরে একটি heading যোগ করা সম্ভব, তবে সাধারণত ফাইলটি সংরক্ষণ করার পর ম্যানুয়ালি একটি শীর্ষ সারি যোগ করাই সবচেয়ে সহজ।

## 5. সর্বাধিক অ্যালার্টযুক্ত তারিখ খুঁজে বের করা

Hayabusa ডিফল্টভাবে, severity level অনুসারে সবচেয়ে বেশি অ্যালার্টযুক্ত তারিখগুলো আপনাকে জানাবে।
তবে, আপনি দ্বিতীয়, তৃতীয় ইত্যাদি সবচেয়ে বেশি অ্যালার্টযুক্ত তারিখগুলোও খুঁজে পেতে চাইতে পারেন।
আমরা আপনার প্রয়োজন অনুসারে বছর, মাস বা তারিখ অনুযায়ী গ্রুপ করতে timestamp স্ট্রিং স্লাইসিং করে তা করতে পারি।

উদাহরণ: `cat results.json | jq ' .Timestamp | .[:10] ' -r | sort | uniq -c | sort`

`.[:10]` `jq`-কে `Timestamp` থেকে কেবল প্রথম 10 বাইট বের করতে বলে।

এটি আমাদের সবচেয়ে বেশি ইভেন্টযুক্ত তারিখগুলো দেবে:
```
1066 2021-12-12
1093 2016-09-02
1571 2021-04-22
1750 2016-09-03
2271 2016-08-19
2932 2021-11-03
8095 2016-09-20
```

আপনি যদি সবচেয়ে বেশি ইভেন্টযুক্ত মাসটি জানতে চান, তবে আপনি প্রথম 7 বাইট বের করতে কেবল `.[:10]`-কে `.[:7]`-এ পরিবর্তন করতে পারেন।

আপনি যদি সবচেয়ে বেশি `high` অ্যালার্টযুক্ত তারিখগুলো তালিকাভুক্ত করতে চান, তবে আপনি এটি করতে পারেন:

`cat results.json | jq 'select ( .Level == "high" ) | .Timestamp | .[:10] ' -r | sort | uniq -c | sort`

আপনি আপনার প্রয়োজন অনুসারে computer name, event ID ইত্যাদি অনুযায়ী `select` ফাংশনে ফিল্টার শর্ত যোগ করতে থাকতে পারেন।

## 6. PowerShell লগ পুনর্গঠন

PowerShell লগ সম্পর্কে একটি দুর্ভাগ্যজনক বিষয় হলো লগগুলো প্রায়ই একাধিক লগে বিভক্ত হয়ে যায় যা পড়া কঠিন করে তোলে।
আক্রমণকারী যে কমান্ডগুলো চালিয়েছে কেবল সেগুলো বের করে আমরা লগগুলো পড়া অনেক সহজ করতে পারি।

উদাহরণস্বরূপ, আপনার যদি EID `4104` ScriptBlock লগ থাকে, তবে আপনি একটি সহজে পঠনযোগ্য টাইমলাইন তৈরি করতে কেবল সেই ফিল্ডটি বের করতে পারেন।

## 7. সন্দেহজনক নেটওয়ার্ক সংযোগ খুঁজে বের করা

আপনি প্রথমে নিম্নলিখিত কমান্ড দিয়ে সব টার্গেট IP ঠিকানার একটি তালিকা পেতে পারেন:

`cat results.json | jq 'select ( .Details.TgtIP? ) | .Details.TgtIP ' -r | sort | uniq`

আপনার যদি threat intelligence থাকে, তবে আপনি যাচাই করতে পারেন যে IP ঠিকানাগুলোর কোনোটি ক্ষতিকর বলে পরিচিত কিনা।

আপনি নিম্নলিখিত দিয়ে একটি নির্দিষ্ট টার্গেট IP ঠিকানায় কতবার সংযোগ হয়েছিল তা গণনা করতে পারেন:

`cat results.json | jq 'select ( .Details.TgtIP? ) | .Details.TgtIP ' -r | sort | uniq -c | sort -n`

`TgtIP`-কে `SrcIP`-এ পরিবর্তন করে, আপনি উৎস IP ঠিকানার উপর ভিত্তি করে ক্ষতিকর IP ঠিকানার জন্য একই threat intelligence যাচাই করতে পারেন।

ধরা যাক আপনি দেখলেন যে `93.184.220.29` ক্ষতিকর IP ঠিকানাটি আপনার পরিবেশ থেকে সংযুক্ত হচ্ছে।
আপনি নিম্নলিখিত কোয়েরি দিয়ে সেই ইভেন্টগুলোর বিস্তারিত পেতে পারেন:

`cat results.json | jq 'select ( .Details.TgtIP? == "93.184.220.29" ) '`

## 8. এক্সিকিউটেবল বাইনারি হ্যাশ বের করা

Sysmon EID `1` Process Creation লগে, sysmon-কে বাইনারির হ্যাশ গণনা করতে কনফিগার করা যায়।
নিরাপত্তা বিশ্লেষকরা threat intelligence দিয়ে এই হ্যাশগুলো পরিচিত ক্ষতিকর হ্যাশের সাথে তুলনা করতে পারেন।
আপনি নিম্নলিখিত দিয়ে `Hashes` ফিল্ডটি বের করতে পারেন:

`cat results.json | jq 'select ( .Details.Hashes? ) | .Details.Hashes ' -r`

এটি আপনাকে এইরকম হ্যাশের একটি তালিকা দেবে:

```

MD5=E112A827FAB9F8378C76040187A6F336,SHA256=ED369187681A62247E38D930320F1CD771756D0B7B67072D8EC655EF99E14AEB,IMPHASH=8EEAA9499666119D13B3F44ECD77A729
```

Sysmon সাধারণত `MD5`, `SHA1` এবং `IMPHASH`-এর মতো একাধিক হ্যাশ গণনা করবে।
আপনি `jq`-তে regular expression দিয়ে এই হ্যাশগুলো বের করতে পারেন বা উন্নত পারফরম্যান্সের জন্য কেবল string splicing ব্যবহার করতে পারেন।

`cat results.json | jq 'select ( .Details.Hashes? ) | .Details.Hashes | .[4:36] ' -r | sort | uniq`

## 9. PowerShell লগ বের করা

PowerShell Scriptblock লগ (EID: 4104) সাধারণত অনেক লগে বিভক্ত হয় এবং CSV ফরম্যাটে আউটপুট করার সময়, আউটপুট আরও সংক্ষিপ্ত করতে Hayabusa ট্যাব এবং return অক্ষর মুছে দেবে।
তবে, মূল ট্যাব এবং return অক্ষর ফরম্যাটিংসহ এবং লগগুলো একসাথে সংযুক্ত করে powershell লগ বিশ্লেষণ করা সবচেয়ে সহজ।
এখানে `COMPUTER-A` থেকে PowerShell EID 4104 লগ বের করে এবং VSCode ইত্যাদিতে খুলে বিশ্লেষণ করার জন্য একটি `.ps1` ফাইলে সংরক্ষণ করার একটি উদাহরণ দেওয়া হলো।
ScriptBlock ফিল্ডটি বের করার পর, আমরা `\r\n` এবং `\n`-কে return অক্ষর এবং `\t`-কে ট্যাব দিয়ে প্রতিস্থাপন করতে `awk` ব্যবহার করি।

```
cat results.json | jq 'select ( .EventID == 4104 and .Details.ScriptBlock? != "n/a"  and .Computer == "COMPUTER-A.domain.local" ) | .Details.ScriptBlock , "\r\n"' -j | awk '{ gsub(/\\r\\n/,"\r\n"); print; }' | awk '{ gsub(/\\t/, "\t"); print; }' | awk '{ gsub(/\\n/, "\r\n"); print; }' > 4104-PowerShell-Logs.ps1
```

বিশ্লেষক ক্ষতিকর PowerShell কমান্ডের জন্য লগ বিশ্লেষণ করার পর, তাদের সাধারণত সেই কমান্ডগুলো কখন চালানো হয়েছিল তা খুঁজে বের করতে হবে।
এখানে একটি কমান্ড কখন চালানো হয়েছিল তা খুঁজে বের করার জন্য Timestamp এবং PowerShell লগ একটি CSV ফাইলে আউটপুট করার একটি উদাহরণ দেওয়া হলো:

```
cat results.json | jq ' select (.EventID == 4104 and .Details.ScriptBlock? != "n/a" and .Computer == "COMPUTER-A.domain.local") | .Timestamp, ",¦", .Details.ScriptBlock?, "¦\r\n" ' -j | awk '{ gsub(/\\r\\n/,"\r\n"); print; }' | awk '{ gsub(/\\t/,"\t"); print; }' | awk '{ gsub(/\\n/,"\r\n"); print; }' > 4104-PowerShell-Logs.csv
```

নোট: ব্যবহৃত স্ট্রিং ডিলিমিটার হলো `¦` কারণ একক ও ডাবল কোট প্রায়ই PowerShell লগে পাওয়া যায় এবং CSV আউটপুট নষ্ট করবে।
আপনি যখন CSV ফাইল আমদানি করবেন, তখন আপনাকে অ্যাপ্লিকেশনে `¦` স্ট্রিং ডিলিমিটারটি নির্দিষ্ট করতে হবে।
